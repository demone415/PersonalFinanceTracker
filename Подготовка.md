# Тема: Трекер личных финансов с поддержкой электронных чеков

### Требования

**Функциональные:**
- Юзер может добавлять трату\начисление (accrual):
    - путем ручного ввода
    - сканом QR кода с физического чека - тогда с данных qr кода создается начисление, а состав чека подгружается фоном.
- Акруалы имеют базовые поля и чек со списком позиций (отдельные сущности привязанные к акруалу)
- В Акруале не обязательно должен быть чек со списком позиций
- CRUD основных сущностей с валидацией на клиенте и сервере
- Поиск и фильтрация по ключевым полям
- Дашборд с визуализацией данных (графики / статистика)
- Пагинация списков
- Адаптивная вёрстка (desktop + mobile)
- Акруал имеет следующие поля: 
    - Сумма (обязательно)
    - Дата (обязательно)
    - Категория
    - Описание
    - Тип (обязательно)
    - Валюта (рубли по дефолту) (обязательно)
    - Включать ли в статистику\отчеты (флаг) (обязательно)
    - Id для группировки акруалов (бывает, что покупка разбивается на два чека. Например, доставка + продукты) (Guid)
    - Теги (коллекция)
    - Чек
- Чек имеет следующие поля:
    - ФД (Фискальный документ)
    - ФН (Фискальный номер)
    - ФПД (Фискальный признак)
    - Сумма в копейках (обязательно)
    - Дата (обязательно)
    - Номер чека во внешней системе
    - Номер смены
    - ИНН
    - Кассир (фио)
    - Организация
    - Адрес
    - Вид налогообложения
    - Метаданные чека (модель подсмотреть в апи провайдера)
    - Список позиций (коллекция)
- Позиция чека имеет следующие поля:
    - Название товара/услуги (обязательно)
    - Цена (обязательно)
    - Количество (обязательно)
    - Сумма (обязательно)
    - Категории (коллекция)
- Типы акруала: Приход, Возврат прихода, Расход, Возврат расхода
- Получение состава чека из QR кода фискального чека с помощью провайдера эл чеков
- Справочник категорий (иконки, цвета) с возможностью CRUD
- Месячные бюджеты: установка лимита на категорию, прогресс‑бар расхода
- Дашборд: круговая диаграмма расходов по категориям, линейный график динамики за 6 месяцев, топ‑5 категорий
- Фильтрация транзакций по дате, категории, сумме (от/до)
- Импорт транзакций из json‑файла (выгрузка чеков из приложения ФНС "Налоги ФЛ")
- Экспорт отфильтрованных транзакций в CSV
- Мультивалютность: выбор основной валюты, хранение курса на момент транзакции
- Регистрация / авторизация (JWT), два предзаполненных аккаунта (например, личный и семейный); каждый пользователь видит только свои транзакции и бюджеты
- Лог изменений: каждое создание, редактирование и удаление транзакции или бюджета фиксируется с указанием пользователя, времени и значений до/после; журнал доступен для просмотра
- Запуск через `docker compose up`
- Report.md - отчёт о процессе разработки - использованные инструменты, промты, проблемы, которые были и решения.
- Swagger для бекенда в виде спец страницы.


**Нефункциональные:**
- Seed‑данные: 200+ транзакций за 6 месяцев, 12 категорий, 3 бюджета, 2 пользователя
- REST API с OpenAPI/Swagger документацией (в виде живой веб-страницы, а не доки в проекте)
- Не менее 10 unit/integration тестов
- Всё приложение запускается одной командой `docker compose up` (backend + frontend + БД)
- файл README.md инструкцией по запуску (формат - markdown)

### MVP

- Юзер может входить в аккаунт и выходить
- Юзер может руками добавлять акруалы и позиции чека к ним с категориями
- Юзер может смотреть, изменять и добавлять категории

### Подход к разработке

#### Общие

- Нужно декомпозировать проект:
    - Сначала разбить проект на большие куски (Epic). Первый эпик - MVP с базовым функционалом
    - Потом каждый эпик на несколько Story, описывающую функционал (ценность) внутри эпика. Например, добавление "категорий" в проект.
    - Каждую сторю поделить на таски. Каждая таска должна быть достаточно маленькой, чтобы можно было выдать ее агенту и он ее сделал без дополнительных вопросов.

#### Backend

- Новые фичи начинай разрабатывать с тестов (TDD)
- Для типов коллекций моделях используй `ICollection<T>` и `IReadOnlyCollection<T>`, где применимо
- Для дат используй `DateTimeOffset`
- Коллекции не должны быть `null`. Всегда инициализируй их как `[]`
- Создавай богатые модели вместо анемичных
- Используй паттерн `UnitOfWork` при работе с `DbContext`.
- Не обнажай сущности бд из АПИ. Мапь их в дто
- Не используй библиотеки для мапинга
- Используй `Directory.Build.props` и `Directory.Packages.props`
- Используй формат решения `.slnx`
- Для логов примени `Serilog`. Sink - `Console`. Настройка - в appsettings.json
- Используй `FusionCache` c `Redis` backpane для кеширования
- Тесты пиши с помощью `xUnit` и `Moq`
- Для идемпотентных операций используй idempotency-key
- Для `Refit`-клиентов используй консистентность запросов с помощью `Polly`
- Для `background jobs` используй `Hangfire` с персиcтентностью

### Получение состава чека из QR кода фискального чека с помощью провайдера эл чеков

- Должен быть класс "получатор чеков", который будет абстракцией над провайдерами (которых в теории может быть несколько или они будут заменяться с течением времени)
- Пока что есть только один провайдер - [ПроверкаЧека](https://proverkacheka.com/) - документация api в файле `proverka_cheka_documentation_api.docx`
- У этого провайдера ограничение на 15 запросов в день. Это нужно обязательно учесть


#### Получение чеков в "ПроверкаЧека"

Лимит запросов к API жёстко не задан. Рекомендуем придерживаться ограничений до 15 запросов к API в сутки.

По статусам ошибки:
 
code 0. Чек некорректен. Проверьте правильность реквизитов.
С данными реквизитами чек не корректен - либо QR код неверен, либо фискальный аппарат выдает неверный код.
Здесь идет нарушение законодательства, т.к. такие чеки не легитимны (считай - организация обманывает налоговую и покупателя)
 
code 2. Нет информации по чеку (прочее).
Чек корректен, но данные с фискального аппарата не поступили в БД налоговой службы.
 
code 3. Превышено количество обращений по чеку.
Вы запрашивали более 5 раз информацию по чеку и каждый раз мы не смогли получить информацию с налоговой.
У нас введено ограничение на запрос такой информации, иначе мы можем вычерпать имеющиеся лимиты по запросам в налоговую.
 
Дополнительная информация по code 2:
После обращения к нашему API мы обращаемся за данными в налоговую. При получении данных по чеку от налоговой мы сохраняем их у себя и отдаем их в риал-тайм через API. В случае, если данных по чеку в налоговой нет, то мы запоминаем реквизиты чека (ФН, ФД, ФП) и отдаем через API статус о невозможности получить чек.
Повторный вызов API по не полученным чекам возможен не ранее чем через 10 минут.
Эти ограничения необходимы, чтобы конечный пользователь не сканировал подряд раз за разом чек, а мы не обращались в налоговую за данными, которых у них нет.
 
Для чего это надо? Дело в том, что некоторые чеки даже при нормальном режиме работы могут приходить в налоговую с задержкой до 30 дней. Законодательно такое поведение разрешено, далее фискальный накопитель блокирует кассу, пока она не выйдет на связь со своим ОФД и не передаст информацию по чекам.
 
Информацию по не полученным чекам можно запросить не более 5 раз.
Предлагаем реализовать алгоритм повторного получения чека по одной из предложенных схем:
- схема 1:
1-ый запрос - сейчас, условно время T
2-ой запрос - T + 1 день
3-ий запрос - T + 3 дня
4-ый запрос - T + 10 дней
5-ый запрос - T + 30 дней.
- схема 2:
1-ый запрос - сейчас, условно время T.
2-ой запрос - T + 6 часов
3-ий запрос - T + 1 день
4-ый запрос - T + 3 дня
5-ый запрос - T + 10 дней.
 
Ограничений по времени между вызовами нашего API по чекам с разными реквизитами нет. Более того, возможно совершать вызовы API параллельно.
 
Т.о. если вы запросили информацию по не полученному чеку более 5 раз, сервис вернет: {"code":3,"data":"Превышено количество обращений по чеку."}
 
При частых запросах одного и того же неполученного чека (менее 10 минут), сервис вернет: {"code":4,"data":"Не вышло время ожидания перед повторным запросом."}
